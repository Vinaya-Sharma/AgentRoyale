Live Web Ground Truth

Tasks are graded against values fetched from the public web, not static answer keys models can memorize.

02

Built To Catch Real Agent Failures

The benchmark tests the boring but costly mistakes: stale data, wrong sources, bad citations, wrong regions, and incorrect product variants.

These are the failures that make AI agents look useful in a demo and break when the answer has to be exact.

03

01

Why I Ran It

I’ve been using AI more and more to find real data quickly: prices, counts, rates, facts, and sources I do not want to manually dig through.

I wanted to know which model was actually best at this. When I ran the experiment, the answer was, sadly, none of them without verification.

02

How I Ran It

I designed 43 tasks that require exact, current, source-specific values. I ran 12 model stacks on each task three times.

Then I compared every answer to ground truth fetched independently from the required source and logged the result.

03

What I Found

Across 1,548 scored attempts, the models were right 50% of the time, wrong 38% of the time, and returned no usable answer 13% of the time.

The uncomfortable part is that 38% of the time, I was returned false information that looked normal and cited.

How I ran this experiment

I wanted a task bank that forced live retrieval, not memorization. So each question is normal enough for a person to ask, but specific enough to have one deterministic answer.

01

Task Bank Creation

Each question has a required source, field, and grading rule. The answers change over time, so models cannot rely on memorized static answer keys.

Examples: “Using Yahoo Finance, what is NVDA’s current regular-market quote price in USD?” or “On the Hacker News front page right now, what is the title of the #1 ranked story?”

02

Model Selection

I chose a mix of consumer flagships, search-first models, efficient models, and open-weight-style stacks.

Each stack ran the same task bank three times so I could see whether it was consistently correct or just occasionally lucky.

03

Grading

I normalized answers and applied task-specific grading rules before scoring each run for exact accuracy.

I did not use an LLM judge because that would add another layer of subjective interpretation. A judge model might reward a plausible explanation, forgive an approximate value, or miss that the model answered with the wrong field. For this benchmark, I wanted the grading to be stricter: the extracted claim either matches the independently fetched value or it does not.

04

Additional Metrics

I logged source matching, citation behavior, cost, latency, and consistency across repeated runs.

Those metrics help builders make decisions based on what they value: accuracy, speed, price, reliability, or lowest risk of false information.

05

Keep In Mind

This is a v1 slice, not a universal ranking of every model forever. The public task bank has 43 questions, and the domain mix is uneven.

My goal is to run another experiment within the next week with a larger task bank but smaller contestant stack: Grok 4, Sonar Pro Search, Gemini Pro, Nemotron 3 Super, DeepSeek V4 Flash, GPT-4o, Claude Sonnet, and Gemini Flash Lite.

The Arena

Compare two anonymous correct answers and choose the route that felt more direct, reliable, and human.

How to vote

Pick the path that looks more direct, reliable, and source-appropriate. Do not simply vote for fewer citations.

These guidelines are only direction. At the end of the day, vote with your human intuition: which route felt more optimal for getting to the answer?

  • Better: canonical source used early, relevant URLs, clear extraction.
  • Worse: stale articles, third-party distractors, wrong region/SKU, source hopping.
  • Tie: both paths use similarly trustworthy sources with similar focus.

Better: Apple official buy page → base MacBook Air price.

Worse: Apple page plus Walmart, blogs, old newsroom posts, or education pricing for the same answer.

An ideal route is the route a careful human would naturally take for this task.

V1 methodology

How I ran the v1 benchmark

I built Agent Royale to test one narrow question: can AI search stacks retrieve exact, current values from the public web when the answer has to match a specific source?

I made the task rubric fixed, but the answer live. Each task has a required source, field, tolerance, and grading rule. The model gets a normal source-specific question, but it does not get the saved answer. Separately, I fetch ground truth from the required source using Bright Data or a stable public API.

I did this because most real retrieval failures are boring but expensive: stale prices, wrong quote fields, company-size ranges instead of employee counts, wrong regions, wrong product variants, and citations that look right but do not support the answer.

TL;DR

  • I tested 12 model stacks on 43 audited tasks, 3 runs per model per task.
  • That produced 1,548 scored attempts: 770 correct, 584 wrong values, and 194 no-answer responses.
  • The tested stacks averaged 50% exact accuracy. Grok 4 led this slice at 73%.
  • I did not use an LLM judge. The extracted claim either matched the independently fetched source value or it did not.

Read this result carefully

This is a v1 slice, not a universal ranking of every model forever. The public task bank has 43 questions, the domain mix is uneven, and the leaderboard is a frozen snapshot from one run.

The most important limitation is timing: the model calls ran in about 2 hours and 11 minutes, but many saved ground-truth values were fetched earlier. In v2, I will refresh ground truth immediately before each task's model calls.

1. Task Design

I wanted normal questions with deterministic answers.

Each task is written like a real user question, but constrained enough to have one answer I can verify. The source and field matter. “What is NVDA’s price?” is too loose. “Using Yahoo Finance, what is NVDA’s current regular-market quote price in USD?” is gradeable.

I chose dynamic web facts so the model has to engage in retrieval instead of relying on a memorized static answer key. The candidate pool included finance, ecommerce, app stores, LinkedIn/company metrics, social metrics, developer tools, subscriptions, real estate, travel, and business intelligence.

For the public v1 launch, I only show tasks where every model had complete coverage: 3 clean runs for each of the 12 model stacks. That left 43 tasks in the frozen slice.

2. Ground Truth

The answer key was fetched independently from the models.

The model being graded never produces the answer key. For each admitted task, I fetched ground truth separately from the required source using Bright Data or a stable public API, then saved the value, source URL, timestamp, evidence, and grading rule.

If I could not reliably extract the value, I quarantined the task so it did not affect scores. I would rather ship a smaller task bank than include rows where the expected answer is questionable.

This is why the Task Bank page matters. It is the evidence ledger: question, saved answer, source, fetch time, evidence, and grading rule.

3. Model Selection

I tested practical stacks people might actually choose.

I tested a mix of consumer flagships, search-first models, efficient/value models, open-weight-style stacks, and advanced reasoning models. The goal was not to crown a permanent winner. It was to see how different retrieval stacks behave on the same exact task bank.

4. Run Protocol

Same task bank, same repetitions, stored raw outputs.

Each model stack answered each task 3 times. The runner stored the raw answer, extracted claim, citations returned by the provider, latency, estimated cost, and final grading result.

task prompt -> model + web retrieval -> raw answer + citations -> claim extraction -> deterministic grading -> stored run

By “model stack,” I mean the model plus the retrieval path used in this runner. That matters because builders do not deploy a base model alone; they deploy a model plus search behavior, citations, routing, and provider behavior.

The public model run window was May 20, 2026, 1:48-3:59 AM PT.

5. Scoring

Exact matching, not vibes.

The main score is Live Exact Accuracy: correct scored runs divided by total scored runs. No-answer runs are included in the denominator because they still failed to return the requested value.

I did not use an LLM judge because that would add another layer of subjective interpretation. A judge model might reward a plausible explanation, forgive an approximate value, or miss that the model answered with the wrong field. For this benchmark, I wanted stricter grading: the extracted claim either matched the independently fetched value or it did not.

I split failures into two buckets: Wrong Answer, where the model returned a usable but incorrect value, and No Answer, where it refused, could not verify, or returned nothing usable. This distinction matters because confident false information is a different product risk than refusal.

6. Metrics I Report

Accuracy alone is not enough for stack decisions.

I report exact accuracy, wrong-answer rate, no-answer rate, canonical-source citation rate, latency, cost, and consistency across repeated runs.

Canonical Source is intentionally limited in v1. It means a cited URL overlapped the required source URL, and I only count it among correct answers. It is not passage-level proof that the cited text supports the value.

These metrics are meant to help builders make tradeoffs. A model can be accurate but too slow, cheap but confidently wrong, or good once but inconsistent across repeated attempts.

9. What Changes Next

V2 should be stricter, larger, and more useful for decisions.

In the next version, I want to make the methodology stricter and the product more useful for teams choosing an AI stack.

  1. Refresh ground truth per task immediately before model calls.
  2. Run all models for that task in parallel right after the refresh.
  3. Store ground_truth_value, ground_truth_fetched_at, ground_truth_source_url, and a ground_truth_id on every model run.
  4. Show "ground truth fetched X minutes before model run" on result pages.
  5. Grow the task bank to at least 75 questions with a more balanced split across finance, apps, ecommerce, social, developer metadata, company metrics, and subscriptions.
  6. Rerun a tighter 8-model suite so the leaderboard is easier to interpret and cheaper to repeat.
  7. Add failure labels: wrong source, wrong field, stale value, unit mismatch, unsupported claim, no answer, and provider/API failure.
  8. Test retrieval-native systems directly, including Exa and Parallel.ai, to see whether their search claims hold up on exact source-specific tasks.
  9. Let teams plug in their own recurring task banks so they can choose the best stack for their actual domain.
  10. Add decision views: cheapest acceptable model, fastest accurate model, most reliable model by domain, and safest model when wrong answers are expensive.
  11. Make source verification stronger by resolving redirects and checking whether the cited evidence actually supports the value.

The goal is to build a repeatable way to check whether AI systems can be trusted with facts that change, and to help people choose the right retrieval stack instead of trusting a vague “good at search” claim.
